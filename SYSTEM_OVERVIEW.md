# Proji — Описание системы и взаимодействий

## Что такое Proji?

**Proji** — это корпоративная бизнес-платформа для управления компанией: задачи, проекты, отчёты, CRM, юридический учёт, аналитика и AI-ассистент — всё в одном интерфейсе.

Платформа ориентирована на команды от 5 до 200 человек и позволяет каждому сотруднику работать в своём домене (Маркетинг, Финансы, Юридический и т.д.) с персонализированным набором инструментов.

---

## Роли в системе

| Роль | Доступ |
|------|--------|
| **Администратор** | Полный доступ ко всем разделам, рассылка задач, управленческие отчёты |
| **Сотрудник** | Работа в своём домене, получение задач, отправка отчётов |

---

## Домены (отделы)

Система работает в 9 доменах, каждый со своим AI-консультантом и набором инструментов:

| Домен | Консультант | Ключевые инструменты |
|-------|------------|----------------------|
| Общий | Бизнес-ассистент | Задачи, заметки, аналитика |
| Финансы | CFO | P&L, бюджет, прогнозы |
| Маркетинг | CMO | Кампании, лиды (CRM), SEO |
| Стратегия | Стратег | OKR, дорожная карта, HADI |
| Операции | COO | Процессы, ресурсы, логистика |
| Юридический | Главный юрисконсульт | Договоры, судебные дела |
| Управление | CEO | Управленческий отчёт, журнал |
| Производство | CTO/Plant Manager | TQM, агайл, регламенты |
| Оборудование | Главный механик | Журнал, схемы, инспекции |

## Основные сущности и их взаимосвязи

```
Пользователь (User)
    │
    ├── принадлежит Домену (BusinessDomain)
    │
    ├── создаёт → Проекты (Project)
    │                │
    │                ├── содержит → Задачи (ProjectTask)
    │                │                 │
    │                │                 └── статус: new → accepted → completed
    │                │
    │                ├── содержит → Отчёты (Report)
    │                │
    │                └── содержит → Документы (Doc)
    │
    ├── получает → Задачи администратора (AdminTask)
    │              (фильтруются по домену пользователя)
    │
    └── пишет → Записи журнала (JournalNote)
```

## Пользовательские сценарии

### Сценарий 1: Сотрудник получает и выполняет задачу

```
1. Администратор создаёт задачу на странице /admin
   → выбирает отдел «Маркетинг», приоритет «Высокий», срок

2. Сотрудник открывает страницу своего домена (Маркетинг)
   → видит задачу в разделе «Задачи от руководства»

3. Сотрудник переходит в проект → Задачи
   → видит свои рабочие задачи
   → кликает на задачу → открывается модальное окно

4. Принимает задачу (статус: new → accepted)

5. Выполняет работу, нажимает «Завершить успешно» (статус: accepted → completed)

6. Переходит в Отчёты проекта
   → выполненная задача появляется в списке
   → включает таймер (фиксирует время работы)
   → выбирает задачу, пишет комментарий
   → нажимает «Отправить отчёт администратору»

7. Отчёт сохраняется и виден администратору
   на странице /management/report
```

### Сценарий 2: Менеджер контролирует команду

```
1. Открывает /management/report
   → видит KPI всех отделов (выручка, NPS, OKR)
   → видит сводку задач по отделам
   → читает последние отчёты сотрудников

2. Замечает проблему → открывает /admin
   → создаёт broadcast-задачу «Срочно: обновить отчётность»
   → выбирает «Финансы», приоритет «Высокий»

3. Сотрудник финансового отдела видит задачу в своём дашборде
```

### Сценарий 3: Юрист ведёт договорную работу

```
1. Открывает /legal/dashboard
   → видит таблицу договоров, фильтрует по «Истёкшим»
   → замечает истёкший лицензионный договор

2. В разделе «Дела» проверяет статус судебных разбирательств

3. В разделе «От руководства» видит задачи от CEO
   → принимает задачу, отмечает выполненной
```

### Сценарий 4: Маркетолог работает с воронкой

```
1. /campaigns — создаёт рекламную кампанию в Google Ads
   → отслеживает CTR и расход бюджета

2. /leads — новые лиды появляются в воронке
   → переключает между Таблицей и Kanban
   → квалифицирует лиды, двигает по воронке

3. /seo — анализирует позиции ключевых слов
   → видит рост по запросам, корректирует контент-план
```

## Поток данных (текущий — localStorage)

```
[Компонент React]
       │
       │ вызывает
       ▼
[src/lib/api.ts]  ← единая точка входа для всех данных
       │
       │ читает/пишет
       ▼
[localStorage]  (ключи: proji_projects, proji_ptasks_*, proji_reports_*, ...)
```

## Поток данных (целевой — REST API)

```
[Компонент React]
       │
       │ вызывает (без изменений в компоненте)
       ▼
[src/lib/api.ts]  ← меняем только тело функций
       │
       │ fetch() с JWT
       ▼
[REST API :4000]
       │
       │ SQL
       ▼
[PostgreSQL]
```

## Хранилище localStorage (ключи)

| Ключ | Содержимое |
|------|-----------|
| `proji_projects` | Массив всех проектов |
| `proji_ptasks_{projectId}` | Задачи конкретного проекта |
| `proji_reports_{projectId}` | Отчёты конкретного проекта |
| `proji_docs_{projectId}` | Документы конкретного проекта |
| `proji_timer_{projectId}` | Состояние таймера |
| `proji_admin_tasks` | Broadcast-задачи администратора |
| `proji_journal_notes_v2` | Записи управленческого журнала |
| `proji_profile` | Профиль текущего пользователя |

## AI-ассистент

- Каждый домен имеет своего AI-консультанта (CFO, CMO, CEO и т.д.)
- Запросы к AI идут через `/api/ai` (Next.js API route → Gemini API)
- AI видит контекст домена и даёт ответы в роли соответствующего специалиста
- История чатов хранится в Zustand (в памяти сессии)

## Технические ограничения (сейчас)

1. **Нет авторизации** — любой пользователь видит все данные в браузере
2. **Данные только в одном браузере** — нет синхронизации между устройствами
3. **Нет многопользовательского доступа** — один браузер = один пользователь
4. **AI без памяти** — история чата сбрасывается при перезагрузке

Все ограничения устраняются после подключения бэкенда (см. `MIGRATION_GUIDE.md`).
